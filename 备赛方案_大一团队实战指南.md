# Agent Hackathon 备赛方案 — 三人团队（大一学生）实战指南

> **赛事：** Agent Hackathon 大赛（AtomGit AI × Ascend 昇腾）
> **团队：** 三人，大一学生，编程基础一般但学习能力强
> **目标：** 交付一个「能跑通商业闭环的行业 Agent」，争取获奖
> **推荐场景：** 广告法合规审查 Agent
> **日期：** 2026.06.18

## 一、场景选型分析

### 选场景的核心框架

大一学生要赢，场景选择必须同时满足五个条件：

1. **一句话能说清** — 评委 5 秒内理解你在解决谁的什么问题
2. **痛点有真实数字** — "省了多少小时/多少钱/避免了多少罚款"
3. **流程可编排** — 业务逻辑是确定性的步骤链，不是靠聊天瞎猜
4. **RAG有料** — 能找到公开的法规、标准、文档喂进去
5. **Demo可跑** — 能用真实数据在台上跑通一轮完整闭环

### 三个候选场景对比

| 维度 | 食安合规巡检 | 跨境评论分析 | **广告法合规审查** ✅ |
|------|------------|------------|-------------------|
| 场景落地性 | 高 | 高 | **极高** — 人人都要写文案 |
| 技术复杂度 | 中（OCR+规则引擎） | 中（爬虫+NLP+聚类） | **中低**（规则+LLM语境） |
| 数据获取难度 | 中（需模拟巡检数据） | 低（评论数据公开可爬） | **低**（《广告法》全文公开） |
| Demo 冲击力 | 高 | 中 | **极高** — 输入文案，秒出标红 |
| 评委理解成本 | 低 | 中 | **极低** — 谁都知道广告法 |
| 可量化程度 | 高 | 高 | **极高** — 罚款金额直接吓人 |
| RAG 素材丰富度 | 高 | 中 | **极高** — 法规+案例+判例 |

### 最终推荐：广告法合规审查 Agent

**一句话定位：** 帮电商运营和自媒体创作者自动审查广告文案是否违反《广告法》，标注违规点并给出合规替代建议。

**推荐理由：**

1. **评委 5 秒就懂** — 不需要解释任何行业背景，"帮运营写安全文案"一句话说清
2. **Demo 效果炸裂** — 输入真实淘宝商品描述，5 秒后全屏标红，给出法规引用和替代建议
3. **数字够吓人** — "一次违规罚款 20 万，我们 10 秒就能帮你查出来"
4. **技术路径最清晰** — 4 个工具各司其职，大一学生可独立完成
5. **评测集最好做** — 违规类型有限，答案客观明确
6. **可扩展故事好讲** — "今天审广告，明天审招聘，后天审合同"

## 二、技术方案

### 推荐技术栈

| 层次 | 选择 | 理由 |
|------|------|------|
| **编程语言** | Python 3.10+ | 唯一选择，生态最全 |
| **Agent 框架** | **LangChain 0.2+** | 文档多、社区大，评分时能讲技术细节 |
| **LLM** | **通义千问 qwen-plus / DeepSeek** | OpenAI 兼容接口，改个 base_url 就行，国产加分 |
| **RAG** | **LangChain + FAISS** | 零配置、纯 Python、不需要装数据库 |
| **Embedding** | text2vec-base-chinese 或调 API | 本地模型不依赖网络，比赛现场稳 |
| **前端** | **Streamlit** | 10 行代码出网页界面，不需要前端知识 |
| **工具调用** | LangChain Tool 装饰器 | 一个装饰器就是一个工具，极简 |
| **结构化输出** | Pydantic + `llm.with_structured_output()` | 保证输出格式稳定 |
| **评测** | Python 脚本 + JSON 题库 | 不需要框架，纯脚本搞定 |
| **版本管理** | Git + GitHub / AtomGit | 比赛标配 |

#### 2. 四个工具定义

```python
from langchain.tools import tool

@tool
def check_prohibited_words(text: str) -> str:
    """
    禁用词规则匹配（第一道筛查）。输入文案文本，基于禁用词库
    进行确定性匹配，返回疑似违规词列表及其初步分类。
    速度极快（毫秒级），作为 LLM 判断之前的预筛。
    """
    matches = []
    for word, info in PROHIBITED_WORDS_DB.items():
        if word in text:
            matches.append({
                "word": word,
                "category": info["category"],     # 极限用语/虚假宣传等
                "severity_hint": info["severity"], # 初步严重程度
            })
    return str(matches) if matches else "未发现禁用词匹配，但建议进一步语境审查"

@tool
def judge_context(text: str, suspected_word: str, category: str) -> str:
    """
    LLM 语境判断（第二道精筛）。输入文案、疑似违规词和初步分类，
    调用 LLM 判断在该语境下是否真正违规。
    用于减少误报（如「最好」出现在「最好是明天来」中不构成广告违规）。
    """
    prompt = f"""你是一位广告法合规审查专家。请判断以下广告文案中的
「{suspected_word}」在该语境下是否构成广告法违规。

文案：{text}
初步分类：{category}

请严格按以下格式回答：
- 是否违规：是/否
- 判断理由：（一句话）
- 置信度：高/中/低"""

    return llm.invoke(prompt)

@tool
def search_law(violation_description: str, industry: str = "通用") -> str:
    """
    法规 RAG 检索（三库并行）。输入违规描述和行业，同时检索：
    - 法规条文库：精确引用法条编号和原文（如「广告法第九条第(三)项」）
    - 处罚案例库：提供同类真实处罚案例和罚款金额
    - 行业规则库：给出该行业的合规替代建议
    返回格式化的知识片段，供 Agent 引用溯源。
    """
    results = retriever.retrieve(violation_description, industry=industry)

    formatted = []
    for doc in results:
        doc_type = doc.metadata.get("doc_type", "unknown")
        if doc_type == "law":
            article = doc.metadata.get("article", "")
            formatted.append(f"📖 【法条 {article}】{doc.page_content}")
        elif doc_type == "case":
            penalty = doc.metadata.get("penalty_amount", "未知")
            formatted.append(f"📋 【处罚案例·罚款{penalty}】{doc.page_content}")
        elif doc_type == "rule":
            formatted.append(f"📌 【行业合规建议】{doc.page_content}")

    return "\n---\n".join(formatted) if formatted else "未检索到相关法规，请人工核查"

@tool
def generate_compliant_text(original_text: str, violations_json: str) -> str:
    """
    替代文案生成（第四步）。输入原文案和违规点列表，
    生成合规的替代文案。不只是诊断，直接给解决方案。
    建议来源于行业规则库中的「禁用→替代」映射。
    """
    prompt = f"""你是一位广告法合规专家。以下文案存在违规，请将其改写为合规版本。
保留原文的核心卖点表达，只修改违规部分。

原文案：{original_text}
违规点：{violations_json}

要求：
1. 保持文案的营销力和吸引力
2. 每处修改标注修改原因
3. 确保改写后不再违反广告法"""

    return llm.invoke(prompt)
```

#### 3. 结构化输出

```python
from pydantic import BaseModel

class ViolationItem(BaseModel):
    position: str           # 违规位置（原文引用）
    violation_type: str     # 违规类型（极限用语/虚假宣传等）
    law_reference: str      # 法条引用（如"广告法第九条第(三)项"）
    risk_level: str         # 风险等级（高/中/低）
    suggestion: str         # 替代建议

class ComplianceReport(BaseModel):
    violations: list[ViolationItem]   # 违规点列表
    overall_risk: str                 # 整体风险等级
    summary: str                      # 审查摘要
    total_violations: int             # 违规总数

llm.with_structured_output(ComplianceReport)
```

#### 4. RAG 知识库架构（核心竞争力，必须做深）

> **为什么 RAG 是这个项目最重要的技术亮点？**
> 评委明确说了：「调数据接口 ≠ 接入知识」。你的 Agent 不能只是调 LLM 然后让它编，必须**真正把行业知识灌进去**，让每一个判断都有据可查。这是区分「套壳 ChatGPT」和「行业 Agent」的分水岭。

##### 4.1 三层知识库设计

不要把所有东西塞进一个向量库——这是最常见的错误。广告法合规审查需要**三个独立的知识子库**，各司其职：

```

│                                                         │
│  ┌─────────────────────────────────────────────────┐    │
│  │  子库 A：法规条文库（精准引用层）                  │    │
│  │  ├── 《广告法》全文 70 条，按条文逐条存储          │    │
│  │  ├── 《广告法实施细则》相关条文                    │    │
│  │  ├── 《互联网广告管理办法》相关条文                │    │
```

##### 4.2 数据来源与采集方案（比赛前就能准备好）

| 知识子库 | 数据来源 | 条数 | 采集方式 |
|---------|---------|------|---------|
| **法规条文库** | 北大法宝 / 国家法律法规数据库 | ~150 条 | 手动复制整理，1小时搞定 |
| **处罚案例库** | 市场监管总局官网 → 行政处罚公示栏 | 50-100 条 | 爬虫或手动摘录关键字段 |
| **行业规则库** | 各行业广告审查指南 + 电商法务公众号文章 | ~200 条 | 人工整理成结构化 JSON |
| **禁用词映射表** | 市监局公开的禁用词清单 + 网上法务整理 | 200+ 对 | GitHub 搜「广告禁用词」直接拿来 |

##### 4.3 文档切分策略（不是无脑切500字）

不同类型的文档必须用不同的切分方式——这是大多数人忽略的关键细节：

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

# ═══════════════════════════════════════════════════════════
# 子库 A：法规条文 —— 按「条」切分，一条就是一chunk
# ═══════════════════════════════════════════════════════════
# 不能用固定长度切！一条法律条文可能100字也可能500字，
# 必须按条文边界切，否则会把一条法律切成两半

class LawArticleSplitter:
    """按法律条文的自然边界切分"""
    def split(self, text: str) -> list[dict]:
        articles = []
        # 匹配「第九条」「第十条第(三)项」等法律条文格式
        import re
        pattern = r'(第[一二三四五六七八九十百]+条(?:第[（(][一二三四五六七八九十]+[）]项)?)\s*(.*?)(?=第[一二三四五六七八九十百]+条|$)'
        matches = re.findall(pattern, text, re.DOTALL)
        for title, content in matches:
            articles.append({
                "page_content": f"{title} {content.strip()}",
                "metadata": {
                    "source": "广告法",
                    "article": title,       # "第九条第(三)项"
                    "doc_type": "law",       # 标记文档类型
                }
            })
        return articles

# ═══════════════════════════════════════════════════════════
# 子库 B：处罚案例 —— 按「案」切分，一案一chunk
# ═══════════════════════════════════════════════════════════
# 一个案例的完整信息（违规内容+处罚结果+法条引用）必须在一起，
# 切散了就没法学习「什么被罚了、为什么被罚、罚了多少」

class CaseSplitter:
    """按行政处罚案例的自然边界切分"""
    def split(self, cases_json: list[dict]) -> list[dict]:
        chunks = []
        for case in cases_json:
            # 每个案例是一个完整的chunk
            content = (
                f"【案例】{case['title']}\n"
                f"【违规内容】{case['violation_text']}\n"
                f"【处罚依据】{case['law_refs']}\n"
                f"【处罚结果】{case['penalty']}\n"
                f"【行业】{case['industry']}\n"
                f"【违规类型】{case['violation_type']}"
            )
            chunks.append({
                "page_content": content,
                "metadata": {
                    "source": case["source_url"],
                    "doc_type": "case",
                    "industry": case["industry"],
                    "violation_type": case["violation_type"],
                    "penalty_amount": case["penalty_amount"],
                }
            })
        return chunks

# ═══════════════════════════════════════════════════════════
# 子库 C：行业规则 —— 按「规则条目」切分
# ═══════════════════════════════════════════════════════════
# 禁用词→替代词的映射要保持完整，不能切成「禁止用最」和「好用极」

class RuleSplitter:
    """按行业规则条目切分"""
    def split(self, rules_json: list[dict]) -> list[dict]:
        chunks = []
        for rule in rules_json:
            content = (
                f"【行业】{rule['industry']}\n"
                f"【禁用表述】{rule['prohibited']}\n"
                f"【违规原因】{rule['reason']}\n"
                f"【合规替代】{rule['alternatives']}\n"
                f"【参考法条】{rule['law_ref']}"
            )
            chunks.append({
                "page_content": content,
                "metadata": {
                    "doc_type": "rule",
                    "industry": rule["industry"],
                    "violation_type": rule["violation_type"],
                }
            })
        return chunks
```

##### 4.4 Embedding 策略（不是直接扔进去就完了）

```python
from langchain_community.vectorstores import FAISS
from langchain_community.embeddings import HuggingFaceEmbeddings

# ═══════════════════════════════════════════════════════════
# 关键设计：Embedding 前的知识增强
# ═══════════════════════════════════════════════════════════
# 直接 embedding 法条原文，效果会很差。因为用户问的是
# "全网最低价算不算违规"，而法条写的是
# "不得使用国家级、最高级、最佳等用语"——语义距离很远。
#
# 解决方案：在 embedding 之前，给每个 chunk 加上
# 「查询增强前缀」，把法条语言翻译成用户会问的语言。

def enrich_for_embedding(chunk: dict) -> str:
    """在 embedding 前，给 chunk 加上查询增强前缀"""
    content = chunk["page_content"]
    doc_type = chunk["metadata"]["doc_type"]

    if doc_type == "law":
        # 给法条加上「用户会怎么问」的自然语言描述
        # 这样用户问"能不能说最好"时，能匹配到"最高级"条款
        enhanced = (
            f"广告法法规条文。{content}\n"
            f"本条禁止的常见表述包括：最、第一、顶级、极品、"
            f"国家级、世界级、全网最低、史上最强等极限用语。"
        )
    elif doc_type == "case":
        enhanced = f"广告违规处罚案例。{content}"
    elif doc_type == "rule":
        enhanced = f"行业广告合规规则和替代建议。{content}"
    else:
        enhanced = content

    return enhanced

# ═══════════════════════════════════════════════════════════
# 构建向量库（三个子库分别建，检索时分别查）
# ═══════════════════════════════════════════════════════════

embeddings = HuggingFaceEmbeddings(
    model_name="shibing624/text2vec-base-chinese",
    model_kwargs={"device": "cpu"},       # 比赛现场没GPU也能跑
    encode_kwargs={"normalize_embeddings": True}  # 归一化，余弦相似度更准
)

# 法规条文向量库
law_docs = LawArticleSplitter().split(law_full_text)
law_enriched = [enrich_for_embedding(d) for d in law_docs]
law_vectorstore = FAISS.from_texts(law_enriched, embeddings, metadatas=[d["metadata"] for d in law_docs])

# 处罚案例向量库
case_docs = CaseSplitter().split(cases_json)
case_enriched = [enrich_for_embedding(d) for d in case_docs]
case_vectorstore = FAISS.from_texts(case_enriched, embeddings, metadatas=[d["metadata"] for d in case_docs])

# 行业规则向量库
rule_docs = RuleSplitter().split(rules_json)
rule_enriched = [enrich_for_embedding(d) for d in rule_docs]
rule_vectorstore = FAISS.from_texts(rule_enriched, embeddings, metadatas=[d["metadata"] for d in rule_docs])
```

##### 4.5 检索策略（不是只做 vector search）

```
用户输入："这款面膜效果超级好，用一次就变白"

检索流程（并行查询三个子库 → 合并排序 → 返回）：

┌────────────────────────────────────────────────────────┐
│ Step 1：并行检索三个子库（各取 top-3）                    │
│                                                        │
│ 法规库 → [广告法第九条(三), 广告法第四条, 化妆品条例第15条] │
│ 案例库 → [某面膜虚假宣传案, 某美白产品处罚案, ...]        │
│ 规则库 → [化妆品禁用词清单, 美白类宣称规则, ...]         │
│                                                        │
│ Step 2：按 metadata 过滤（可选）                         │
│ → 如果能判断文案属于「化妆品」行业                        │
│ → 优先返回 industry=化妆品 的规则和案例                   │
│                                                        │
│ Step 3：合并去重，按相似度排序，取 top-5                  │
│                                                        │
│ Step 4：返回给 Agent 作为上下文                          │
└────────────────────────────────────────────────────────┘
```

```python
from langchain.schema import Document
from typing import Optional

class AdComplianceRetriever:
    """广告法合规专用检索器 —— 三库并行 + 行业过滤"""

    def __init__(self):
        self.law_retriever = law_vectorstore.as_retriever(
            search_kwargs={"k": 3}
        )
        self.case_retriever = case_vectorstore.as_retriever(
            search_kwargs={"k": 3}
        )
        self.rule_retriever = rule_vectorstore.as_retriever(
            search_kwargs={"k": 3}
        )

    def retrieve(
        self,
        query: str,
        industry: Optional[str] = None
    ) -> list[Document]:
        """
        并行检索三个子库，返回合并后的 top 结果。

        Args:
            query: 用户查询（可以是文案片段、违规类型等）
            industry: 行业标签（可选，用于过滤）
        Returns:
            合并后的 Document 列表，按相关性排序
        """
        # 并行检索三个子库
        law_results = self.law_retriever.invoke(query)
        case_results = self.case_retriever.invoke(query)
        rule_results = self.rule_retriever.invoke(query)

        all_results = law_results + case_results + rule_results

        # 如果知道行业，提升同行业结果的权重
        if industry:
            industry_boost = [
                r for r in all_results
                if r.metadata.get("industry") == industry
            ]
            other = [
                r for r in all_results
                if r.metadata.get("industry") != industry
            ]
            # 同行业结果排前面
            all_results = industry_boost + other

        # 去重（同一个法条不要出现两次）
        seen = set()
        unique_results = []
        for doc in all_results:
            key = doc.page_content[:100]  # 用前100字去重
            if key not in seen:
                seen.add(key)
                unique_results.append(doc)

        return unique_results[:5]  # 返回 top-5

# ═══════════════════════════════════════════════════════════
# 在 Agent 的 tool 中调用
# ═══════════════════════════════════════════════════════════

retriever = AdComplianceRetriever()

@tool
def search_law(violation_description: str, industry: str = "通用") -> str:
    """
    法规 RAG 检索。输入违规描述（如「使用极限用语」）和行业，
    返回相关法条原文、类似处罚案例、以及合规替代建议。

    内部同时检索三个知识子库：
    - 法规条文库：精确引用法条编号和原文
    - 处罚案例库：提供真实处罚案例作为参考
    - 行业规则库：给出该行业的具体合规建议
    """
    results = retriever.retrieve(violation_description, industry=industry)

    formatted = []
    for i, doc in enumerate(results):
        doc_type = doc.metadata.get("doc_type", "unknown")
        if doc_type == "law":
            formatted.append(f"📖 法条：{doc.page_content}")
        elif doc_type == "case":
            formatted.append(f"📋 案例：{doc.page_content}")
        elif doc_type == "rule":
            formatted.append(f"📌 规则：{doc.page_content}")

    return "\n---\n".join(formatted)
```

##### 4.6 RAG 质量保障（评测时的加分项）

光搭了 RAG 不够，评委会问：「你怎么知道检索出来的内容是对的？」——所以要对 RAG 本身做评测。

```python
# RAG 评测集（独立于 Agent 评测集）
rag_test_cases = [
    {
        "query": "全网最低价算违规吗",
        "expected_law_articles": ["第九条第(三)项"],
        "expected_retrieval_contains": ["最高级", "最佳"],
        "category": "极限用语"
    },
    {
        "query": "化妆品广告说能祛斑美白",
        "expected_law_articles": ["第十七条", "化妆品监督管理条例相关条文"],
        "expected_retrieval_contains": ["化妆品", "功效"],
        "category": "虚假宣传"
    },
    # ... 15-20 题
]

# 评测指标
def evaluate_rag(test_cases):
    recall_hits = 0    # 期望的法条有没有被检索到
    precision_hits = 0 # 检索出来的结果是否相关

    for case in test_cases:
        results = retriever.retrieve(case["query"])
        retrieved_text = " ".join([r.page_content for r in results])

        # Recall：期望的法条是否出现在检索结果中
        expected_found = all(
            keyword in retrieved_text
            for keyword in case["expected_retrieval_contains"]
        )
        recall_hits += int(expected_found)

    recall = recall_hits / len(test_cases)
    print(f"RAG 召回率：{recall * 100:.1f}%")
    # 目标 ≥ 80%——评委看到这个数字会觉得你们认真做了

evaluate_rag(rag_test_cases)
```

##### 4.7 为什么这个 RAG 方案能拿高分

| 维度 | 普通做法（廉价） | 我们的做法（加分） | 评委怎么看 |
|------|----------------|-------------------|-----------|
| 知识库结构 | 所有文档塞一个 FAISS | **三个子库分层**（法规/案例/规则） | "懂行业知识管理" |
| 切分策略 | `chunk_size=500` 无脑切 | **按法律条文/案例/规则条目自然边界切** | "知道文档结构" |
| Embedding | 直接嵌入原文 | **查询增强前缀**（翻译法条语言为用户语言） | "理解语义鸿沟" |
| 检索方式 | 单库 `similarity_search` | **三库并行 + 行业过滤 + 去重排序** | "检索工程能力强" |
| 质量保障 | 不评测 RAG | **RAG 评测集 + 召回率指标** | "有评测意识" |
| 数据溯源 | 不标注来源 | **每个结果标注来源类型+法条编号** | "可靠性工程到位" |

> **答辩时的一句话：** "我们的 RAG 不是简单地把《广告法》扔进向量库。我们把知识拆成法规条文、处罚案例、行业规则三个子库，用不同的切分策略保证知识完整性，检索时三库并行并按行业过滤。这是行业 Agent 区别于通用 ChatGPT 的关键。"

这段话配上架构图一起讲，技术分直接拉满。

## 三、可靠性工程（三道防线）

### 防线 1：溯源

每条违规标注对应法条编号，在输出末尾附上 `[来源: 广告法第九条第(三)项]`。

### 防线 2：结构化输出

用 Pydantic 模型约束输出，不允许自由文本输出关键信息。

### 防线 3：人在回路

Streamlit 界面加一个"确认/修改"按钮，Agent 生成审查报告后暂停，用户确认后才最终输出。特别是"疑似违规"项，需要人判断语境。

## 四、评测方案

### 4.1 Agent 评测集（30 题）

```
题型分布：
├── 10 题 应违规（涵盖所有违规类型）
│   ├── 3 题 极限用语（「全网最低」「国家级」「第一」）
│   ├── 3 题 虚假宣传（「用一次就变白」「包治百病」）
│   ├── 2 题 对比贬低（「比XX品牌好10倍」）
│   └── 2 题 违背社会良好风尚（低俗暗示等）
├── 10 题 不应违规（正样本，测试是否误判）
│   ├── 3 题 主观评价（「味道不错」「手感好」）
│   ├── 3 题 事实陈述（「成立于2010年」「获得XX认证」）
│   └── 4 题 合规营销（「限时优惠」「新品上市」）
└── 10 题 边界case（需要 LLM 判断语境的灰色地带）
    ├── 3 题 同一个词在不同语境（「最好」=违规 vs「最好是提前预约」=不违规）
    ├── 3 题 行业特殊规则（化妆品「美白」vs 食品「美白」）
    └── 4 题 复合违规（一句话里有多个违规点）

每题包含：
- 输入文案
- 是否应违规（ground truth）
- 违规了哪条法条（如果应违规）
- 预期风险等级
- 适用行业标签

评测指标：
├── 精确率 Precision（标注为违规的有多少真是违规）
├── 召回率 Recall（真实违规有多少被找到）
├── F1 分数（精确率和召回率的调和平均）
├── 法条引用准确率（引用的法条是否正确）
└── 整体准确率

目标：F1 ≥ 85%（哪怕不高，有数据就比空话强 100 倍）
```

### 4.2 RAG 评测集（15 题，独立于 Agent 评测）

```
目的：单独验证 RAG 检索质量，不被 Agent 推理干扰。

每题包含：
- 查询语句（模拟用户/Agent 会问的问题）
- 期望检索到的法条编号
- 期望检索结果中必须包含的关键词
- 所属子库（法规/案例/规则）

评测指标：
├── 法规召回率（期望的法条有没有被检索到）
├── 案例相关性（检索到的案例是否和违规类型匹配）
├── 规则覆盖率（是否返回了对应的合规替代建议）
└── 三库命中率（每个查询是否命中了至少两个子库）

目标：法规召回率 ≥ 80%
```

### 4.3 自动评测脚本

```python
import json

# ═══════════════════════════════════════════════════════════
# Agent 评测
# ═══════════════════════════════════════════════════════════

with open("test_cases_agent.json", "r", encoding="utf-8") as f:
    test_cases = json.load(f)

results = {"TP": 0, "FP": 0, "FN": 0, "TN": 0, "law_correct": 0}

for case in test_cases:
    response = agent.invoke(case["text"])
    predicted_violation = response.total_violations > 0
    actual_violation = case["expected_violation"]

    if predicted_violation and actual_violation:
        results["TP"] += 1
        # 检查法条引用是否正确
        if case.get("expected_law"):
            predicted_laws = [v.law_reference for v in response.violations]
            if any(case["expected_law"] in law for law in predicted_laws):
                results["law_correct"] += 1
    elif predicted_violation and not actual_violation:
        results["FP"] += 1
    elif not predicted_violation and actual_violation:
        results["FN"] += 1
    else:
        results["TN"] += 1

precision = results["TP"] / (results["TP"] + results["FP"]) if (results["TP"] + results["FP"]) > 0 else 0
recall = results["TP"] / (results["TP"] + results["FN"]) if (results["TP"] + results["FN"]) > 0 else 0
f1 = 2 * precision * recall / (precision + recall) if (precision + recall) > 0 else 0
law_acc = results["law_correct"] / results["TP"] if results["TP"] > 0 else 0

print(f"精确率: {precision:.1%}")
print(f"召回率: {recall:.1%}")
print(f"F1 分数: {f1:.1%}")
print(f"法条准确率: {law_acc:.1%}")

# ═══════════════════════════════════════════════════════════
# RAG 评测
# ═══════════════════════════════════════════════════════════

with open("test_cases_rag.json", "r", encoding="utf-8") as f:
    rag_cases = json.load(f)

rag_hits = 0
for case in rag_cases:
    results = retriever.retrieve(case["query"])
    retrieved_text = " ".join([r.page_content for r in results])
    if all(kw in retrieved_text for kw in case["expected_keywords"]):
        rag_hits += 1

rag_recall = rag_hits / len(rag_cases)
print(f"\nRAG 召回率: {rag_recall:.1%}")
```
