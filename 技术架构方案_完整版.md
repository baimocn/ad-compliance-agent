## ## 后端架构设计

# 广告法合规审查 Agent — 后端架构设计

## 1. Agent 推理策略：Function Calling（非 ReAct）

**选择理由：**

- ReAct 的 thought-action-observation 循环会产生大量冗余 token，每次推理都要复述上下文。广告法合规审查是确定性流程（先检查禁用词、再语境判断、再查法规、最后生成替代），不需要 Agent 自由探索。
- Function Calling 让模型直接选择工具并输出结构化参数，减少一轮中间推理，延迟降低约 40%。
- DeepSeek-V4-flash 原生支持 OpenAI function calling 协议，tool_choice 可控。

**但有一个关键点：** 我们不完全依赖模型的 tool_choice 路由。禁用词规则匹配是纯确定性逻辑，不需要 LLM 参与决策——它应该在 Agent 调用链之前作为预处理步骤硬编码执行。LLM 只负责语境判断、法规检索决策和替代文案生成这三个需要推理的步骤。

### schemas.py — 结构化输出定义

```python
from pydantic import BaseModel, Field
from typing import Optional
from enum import Enum


class RiskLevel(str, Enum):
    HIGH = "high"        # 明确违反广告法
    MEDIUM = "medium"    # 存在风险，需人工复核
    LOW = "low"          # 基本合规
    PASS = "pass"        # 完全合规


class BannedWordHit(BaseModel):
    """单条禁用词命中记录"""
    word: str = Field(description="命中的禁用词原文")
    category: str = Field(description="禁用词类别：绝对化用语/虚假宣传/违规承诺/迷信/其他")
    position: tuple[int, int] = Field(description="在原文中的起止位置 (start, end)")
    regulation_ref: str = Field(description="对应的法规条款编号，如 '广告法第九条第三项'")


class ContextJudgment(BaseModel):
    """LLM 语境判断结果"""
    is_violation: bool = Field(description="该词在此语境下是否构成违规")
    reasoning: str = Field(description="判断理由，100字以内")
    confidence: float = Field(description="置信度 0-1", ge=0, le=1)


class RegulationCitation(BaseModel):
    """法规引用"""
    law_name: str = Field(description="法规名称，如 '中华人民共和国广告法'")
    article: str = Field(description="条款编号")
    content: str = Field(description="条款原文摘要")
    relevance_score: float = Field(description="与当前文案的相关度")


class RewriteOption(BaseModel):
    """替代文案选项"""
    text: str = Field(description="改写后的文案")
    risk_level: RiskLevel = Field(description="改写后的风险等级")
    changes: list[str] = Field(description="具体修改说明")
    preserved_intent: str = Field(description="保留的原始营销意图")


class ComplianceReviewResult(BaseModel):
    """最终审查结果 — 结构化输出的顶层 Schema"""
    original_text: str = Field(description="原始文案")
    risk_level: RiskLevel = Field(description="整体风险等级")
    banned_word_hits: list[BannedWordHit] = Field(
        default_factory=list,
        description="禁用词命中列表"
    )
    context_judgments: list[ContextJudgment] = Field(
        default_factory=list,
        description="各命中词的语境判断"
    )
    regulation_citations: list[RegulationCitation] = Field(
        default_factory=list,
        description="相关法规引用"
    )
    rewrite_options: list[RewriteOption] = Field(
        default_factory=list,
        description="替代文案选项（仅 risk_level 为 high/medium 时生成）"
    )
    summary: str = Field(description="审查结论摘要，200字以内")
    review_id: str = Field(description="审查单号，格式：CR-{timestamp}-{hash4}")


class ReviewRequest(BaseModel):
    """审查请求"""
    text: str = Field(description="待审查文案", min_length=1, max_length=5000)
    product_category: Optional[str] = Field(
        default=None,
        description="商品类别，用于匹配特定行业禁用词"
    )
    strict_mode: bool = Field(
        default=False,
        description="严格模式：中等风险也判定为违规并生成替代"
    )
```

### tools/banned_word.py — 工具1：禁用词规则匹配

```python
import re
from dataclasses import dataclass
from schemas import BannedWordHit


@dataclass
class BannedWordEntry:
    word: str
    category: str
    regulation_ref: str
    # 正则模式，支持词形变化，如"最"需匹配"最好/最佳/最强"
    pattern: re.Pattern


class BannedWordMatcher:
    """
    确定性规则引擎，不依赖 LLM。
    禁用词表来源：广告法第九条、第十条、第十一条 + 市场监管总局指导案例。
    """

    def __init__(self, dict_path: str = "data/banned_words.json"):
        self._entries: list[BannedWordEntry] = []
        self._load_dictionary(dict_path)

    def _load_dictionary(self, path: str) -> None:
        """加载禁用词典，构建编译后的正则列表"""
        import json
        with open(path, "r", encoding="utf-8") as f:
            raw = json.load(f)
        for item in raw:
            # 对每个词构建正则：支持前后缀通配
            # 例如 "最XX" 模式用正则 r"最[\u4e00-\u9fff]{1,4}"
            pattern = re.compile(item["pattern"], re.IGNORECASE)
            self._entries.append(BannedWordEntry(
                word=item["word"],
                category=item["category"],
                regulation_ref=item["regulation_ref"],
                pattern=pattern,
            ))

    def match(self, text: str) -> list[BannedWordHit]:
        """
        扫描文本，返回所有命中。
        时间复杂度 O(n * m)，n=文本长度，m=词表大小。
        对于 5000 字文案 + 2000 词条的词表，毫秒级完成。
        """
        hits: list[BannedWordHit] = []
        for entry in self._entries:
            for m in entry.pattern.finditer(text):
                hits.append(BannedWordHit(
                    word=m.group(),
                    category=entry.category,
                    position=(m.start(), m.end()),
                    regulation_ref=entry.regulation_ref,
                ))
        # 去重：同一位置只保留第一个命中
        seen_positions: set[tuple[int, int]] = set()
        unique_hits: list[BannedWordHit] = []
        for hit in hits:
            if hit.position not in seen_positions:
                seen_positions.add(hit.position)
                unique_hits.append(hit)
        return unique_hits

    def reload(self) -> None:
        """热更新词表，不停服"""
        self._load_dictionary("data/banned_words.json")
```

**关键设计决策：** 禁用词匹配在 Agent 调用链之外执行，作为预处理。原因是它是纯确定性逻辑（100% 精确匹配），让 LLM 做这件事既浪费 token 又可能遗漏。匹配结果作为上下文传入后续的 LLM 调用。

### tools/context_judge.py — 工具2：LLM 语境判断

```python
from langchain_core.messages import SystemMessage, HumanMessage
from langchain_openai import ChatOpenAI
from schemas import BannedWordHit, ContextJudgment


class ContextJudge:
    """
    对每个禁用词命中做语境判断：
    "全网最低价"中的"最低"是绝对化用语 → 违规。
    "这款产品最低温度可达-20°C"中的"最低"是物理描述 → 不违规。
    """

    def __init__(self, llm: ChatOpenAI):
        # 使用 with_structured_output 强制输出结构化结果
        self._llm = llm.with_structured_output(ContextJudgment)

    async def judge(
        self,
        text: str,
        hit: BannedWordHit,
        product_category: str | None = None,
    ) -> ContextJudgment:
        """
        对单个禁用词命中做语境判断。
        每个 hit 独立调用一次 LLM，确保判断互不干扰。
        """
        system_prompt = """你是广告法合规专家。判断以下禁用词在给定文案语境中是否构成违规。

判断规则：
1. 如果该词用于描述产品功效、价格优势、排名等营销目的 → 违规
2. 如果该词用于客观技术参数、物理属性、科学数据 → 不违规
3. 如果不确定 → 标记为违规（宁可误报不可漏报）

输出要求：
- is_violation: 布尔值
- reasoning: 100字以内的判断理由
- confidence: 0-1 的置信度"""

        category_hint = f"\n商品类别：{product_category}" if product_category else ""

        user_msg = f"""待审查文案：
{text}

命中禁用词："{hit.word}"
禁用词类别：{hit.category}
命中位置：第{hit.position[0]}字到第{hit.position[1]}字{category_hint}"""

        result: ContextJudgment = await self._llm.ainvoke([
            SystemMessage(content=system_prompt),
            HumanMessage(content=user_msg),
        ])
        return result

    async def batch_judge(
        self,
        text: str,
        hits: list[BannedWordHit],
        product_category: str | None = None,
    ) -> list[ContextJudgment]:
        """
        批量判断，使用 asyncio.gather 并发调用。
        对于 N 个命中词，延迟约为单次调用时间（并发）。
        """
        import asyncio
        tasks = [
            self.judge(text, hit, product_category)
            for hit in hits
        ]
        results = await asyncio.gather(*tasks, return_exceptions=True)

        # 异常降级：某个 hit 判断失败时，默认标记为违规
        judgments: list[ContextJudgment] = []
        for i, result in enumerate(results):
            if isinstance(result, Exception):
                judgments.append(ContextJudgment(
                    is_violation=True,  # 降级策略：失败默认违规
                    reasoning=f"语境判断服务异常，默认标记为违规。错误：{str(result)[:80]}",
                    confidence=0.0,
                ))
            else:
                judgments.append(result)
        return judgments
```

### tools/regulation_rag.py — 工具3：法规 RAG 检索

```python
from langchain_core.messages import SystemMessage, HumanMessage
from langchain_openai import ChatOpenAI
from schemas import BannedWordHit, RegulationCitation


class RegulationRAG:
    """
    对接 vmem MemoryVectorStore。
    vmem 底层：SQLite + FTS5 全文检索 + numpy cosine 相似度 + bge-small-zh-v1.5 嵌入。
    这里封装检索 + 重排两步。
    """

    def __init__(self, llm: ChatOpenAI, vmem_client: "VmemClient"):
        self._llm = llm
        self._vmem = vmem_client
        # 法规文档的 namespace，与用户记忆隔离
        self._namespace = "ad_regulations"

    async def index_regulation(self, doc_id: str, content: str, metadata: dict) -> None:
        """
        将法规条文写入 vmem。
        在系统初始化时调用，将广告法、消费者权益保护法等法规按条款切分后逐条写入。
        """
        await self._vmem.store(
            key=f"reg:{doc_id}",
            value=content,
            source="national_regulation",
            level="authoritative",
            tags=["广告法", "法规", metadata.get("law_name", ""), metadata.get("article", "")],
            namespace=self._namespace,
        )

    async def retrieve(
        self,
        text: str,
        hits: list[BannedWordHit],
        top_k: int = 5,
    ) -> list[RegulationCitation]:
        """
        检索相关法规条款。
        查询构造策略：拼接文案 + 命中词 + 类别，形成语义丰富的查询。
        """
        # 构造查询
        categories = list({h.category for h in hits})
        hit_words = list({h.word for h in hits})
        query = (
            f"广告文案合规审查。文案：{text[:200]}。"
            f"涉及禁用词：{'、'.join(hit_words)}。"
            f"类别：{'、'.join(categories)}。"
            f"请检索相关的广告法条款。"
        )

        # vmem 语义检索（bge-small-zh-v1.5 向量相似度）
        results = await self._vmem.search(
            query=query,
            top_k=top_k,
            namespace=self._namespace,
        )

        if not results:
            return []

        # 用 LLM 做重排：从 top_k 结果中筛选最相关的
        citations = await self._rerank(text, hits, results)
        return citations

    async def _rerank(
        self,
        text: str,
        hits: list[BannedWordHit],
        raw_results: list[dict],
    ) -> list[RegulationCitation]:
        """
        LLM 重排：从候选法规中筛选并生成引用。
        使用 structured output 确保格式一致。
        """
        from pydantic import BaseModel

        class RerankOutput(BaseModel):
            citations: list[RegulationCitation]

        rerank_llm = self._llm.with_structured_output(RerankOutput)

        candidates_text = "\n\n".join(
            f"[{i+1}] {r['key']}: {r['value'][:300]}"
            for i, r in enumerate(raw_results)
        )
        hit_summary = "、".join(f'"{h.word}"({h.category})' for h in hits[:5])

        prompt = f"""你是广告法专家。从以下候选法规条款中，筛选与当前审查最相关的条款。

待审查文案：{text[:300]}
命中禁用词：{hit_summary}

候选法规条款：
{candidates_text}

要求：
1. 只保留与文案直接相关的条款（最多5条）
2. 为每条生成 content（条款原文摘要，100字以内）
3. 评估 relevance_score（0-1）"""

        result: RerankOutput = await rerank_llm.ainvoke([
            HumanMessage(content=prompt)
        ])
        return result.citations
```

### tools/rewrite.py — 工具4：替代文案生成

```python
from langchain_core.messages import SystemMessage, HumanMessage
from langchain_openai import ChatOpenAI
from schemas import (
    BannedWordHit, ContextJudgment, RegulationCitation,
    RewriteOption, RiskLevel,
)


class RewriteGenerator:
    """
    基于审查结果生成合规替代文案。
    核心约束：保留营销意图，消除法律风险。
    """

    def __init__(self, llm: ChatOpenAI):
        self._llm = llm

    async def generate(
        self,
        original_text: str,
        violations: list[tuple[BannedWordHit, ContextJudgment]],
        citations: list[RegulationCitation],
        count: int = 3,
    ) -> list[RewriteOption]:
        """
        生成 count 个替代文案选项。

        参数 violations 是 (hit, judgment) 对，只包含被判定为违规的命中。
        不违规的命中不需要修改。
        """
        from pydantic import BaseModel

        class RewriteOutput(BaseModel):
            options: list[RewriteOption]

        rewrite_llm = self._llm.with_structured_output(RewriteOutput)

        # 构造违规详情
        violation_details = []
        for hit, judgment in violations:
            violation_details.append(
                f'- "{hit.word}"（类别：{hit.category}，原因：{judgment.reasoning}）'
            )

        # 构造法规约束
        regulation_constraints = []
        for cite in citations[:3]:  # 最多3条法规约束
            regulation_constraints.append(
                f"- {cite.law_name} {cite.article}：{cite.content[:100]}"
            )

        system_prompt = """你是资深广告文案专家，精通广告法合规要求。

改写原则：
1. 消除所有违规用语，替换为合规表述
2. 保留原始文案的核心营销意图和情感调性
3. 改写后的文案必须朗朗上口，不能像法律文件
4. 每个方案的改写策略应不同（如：数据化表达、场景化描述、对比隐喻等）
5. 不得引入新的合规风险"""

        user_msg = f"""原始文案：
{original_text}

违规项：
{chr(10).join(violation_details)}

相关法规约束：
{chr(10).join(regulation_constraints) if regulation_constraints else "无特定条款"}

请生成 {count} 个合规替代方案。每个方案需说明具体修改内容和保留的营销意图。"""

        result: RewriteOutput = await rewrite_llm.ainvoke([
            SystemMessage(content=system_prompt),
            HumanMessage(content=user_msg),
        ])

        # 对每个方案做自检：用 banned_word_matcher 再跑一遍
        # （在 agent.py 的编排层实现，此处只负责生成）
        return result.options
```

## 3. 工具编排顺序与依赖关系

```
用户输入文案
       │
       ▼
┌─────────────────┐
│  Step 0: 预处理   │  BannedWordMatcher.match(text)
│  禁用词匹配       │  ← 确定性逻辑，不走 LLM
│  (~5ms)          │
└────────┬────────┘
         │
         ▼
    hits 为空？
    ┌────┴────┐
    │ Yes     │ No
    │         │
    ▼         ▼
 返回 PASS   ┌──────────────────────┐
             │  Step 1: 语境判断      │  ContextJudge.batch_judge()
             │  并发评估每个 hit      │  ← LLM call × N（并发）
             │  (~500ms)             │
             └────────┬─────────────┘
                      │
                      ▼
               violations = [is_violation=True 的项]
               violations 为空？
               ┌────┴────┐
               │ Yes     │ No
               │         │
               ▼         ▼
          返回 LOW  ┌──────────────────────┐
                    │  Step 2: 法规检索      │  RegulationRAG.retrieve()
                    │  语义检索 + LLM 重排   │  ← vmem 检索 + LLM call
                    │  (~800ms)              │
                    └────────┬─────────────┘
                             │
                             ▼
                    risk_level = HIGH?
                    ┌────┴────┐
                    │ Yes     │ No (MEDIUM)
                    │         │
                    ▼         ▼
              ┌──────────────────┐  直接输出结果
              │  Step 3: 替代文案  │  RewriteGenerator.generate()
              │  生成 + 自检       │  ← LLM call
              │  (~1000ms)        │
              └────────┬─────────┘
                       │
                       ▼
                  ┌──────────────┐
                  │  Step 4: 自检  │  对每个 rewrite 再跑一遍 banned_word_matcher
                  │  替代文案复检   │  确保不引入新的禁用词
                  │  (~5ms)       │
                  └────────┬──────┘
                           │
                           ▼
                    组装 ComplianceReviewResult
```

**依赖关系总结：**

| 步骤 | 依赖 | 可并发 |
|------|------|--------|
| Step 0 禁用词匹配 | 无 | 独立执行 |
| Step 1 语境判断 | Step 0 的 hits | N 个 hit 之间并发 |
| Step 2 法规检索 | Step 1 的 violations | 内部检索+重排串行 |
| Step 3 替代文案 | Step 1 + Step 2 | 生成多个方案可并发 |
| Step 4 自检 | Step 3 的结果 | 独立执行 |

```python
import uuid

from tools.banned_word import BannedWordMatcher
from tools.context_judge import ContextJudge
from tools.regulation_rag import RegulationRAG
from tools.rewrite import RewriteGenerator
from vmem_client import VmemClient


class ComplianceAgent:
    """
    广告法合规审查 Agent。

    不使用 LangChain AgentExecutor，而是硬编码编排流程。
    原因：广告法审查的工具调用顺序是确定的，不需要 LLM 自主决策调用哪个工具。
    AgentExecutor 的 ReAct 循环在这里是多余的开销。
    """

    def __init__(
        self,
        llm: ChatOpenAI,
        vmem_client: VmemClient,
        dict_path: str = "data/banned_words.json",
    ):
        self._matcher = BannedWordMatcher(dict_path)
        self._judge = ContextJudge(llm)
        self._rag = RegulationRAG(llm, vmem_client)
        self._rewriter = RewriteGenerator(llm)
        self._llm = llm

    async def review(self, request: ReviewRequest) -> ComplianceReviewResult:
        review_id = f"CR-{int(time.time())}-{uuid.uuid4().hex[:4]}"

        # Step 0: 禁用词匹配（确定性，无 LLM）
        hits = self._matcher.match(request.text)

        if not hits:
            return ComplianceReviewResult(
                original_text=request.text,
                risk_level=RiskLevel.PASS,
                banned_word_hits=[],
                context_judgments=[],
                regulation_citations=[],
                rewrite_options=[],
                summary="文案未命中任何禁用词，合规审查通过。",
                review_id=review_id,
            )

        # Step 1: 语境判断（LLM，并发）
        judgments = await self._judge.batch_judge(
            text=request.text,
            hits=hits,
            product_category=request.product_category,
        )

        # 筛选出违规项
        violations = [
            (hit, judgment)
            for hit, judgment in zip(hits, judgments)
            if judgment.is_violation
        ]

        # 如果没有违规项（所有命中词在语境中都不违规）
        if not violations:
            return ComplianceReviewResult(
                original_text=request.text,
                risk_level=RiskLevel.LOW,
                banned_word_hits=hits,
                context_judgments=judgments,
                regulation_citations=[],
                rewrite_options=[],
                summary=f"文案命中 {len(hits)} 个禁用词，但经语境分析均不构成违规。建议人工复核。",
                review_id=review_id,
            )

        # Step 2: 法规检索（vmem + LLM 重排）
        violation_hits = [hit for hit, _ in violations]
        citations = await self._rag.retrieve(
            text=request.text,
            hits=violation_hits,
        )

        # 判定风险等级
        high_confidence_violations = [
            (h, j) for h, j in violations if j.confidence >= 0.8
        ]
        risk_level = (
            RiskLevel.HIGH
            if len(high_confidence_violations) >= 1
            else RiskLevel.MEDIUM
        )

        # 严格模式：MEDIUM 也升级为 HIGH
        if request.strict_mode and risk_level == RiskLevel.MEDIUM:
            risk_level = RiskLevel.HIGH

        # Step 3: 替代文案生成（仅 HIGH 风险时）
        rewrite_options = []
        if risk_level in (RiskLevel.HIGH, RiskLevel.MEDIUM):
            rewrite_options = await self._rewriter.generate(
                original_text=request.text,
                violations=violations,
                citations=citations,
            )

            # Step 4: 自检——替代文案是否引入新禁用词
            rewrite_options = self._self_check_rewrites(rewrite_options)

        # 组装结论
        summary = self._build_summary(
            hits, judgments, violations, citations, risk_level
        )

        return ComplianceReviewResult(
            original_text=request.text,
            risk_level=risk_level,
            banned_word_hits=hits,
            context_judgments=judgments,
            regulation_citations=citations,
            rewrite_options=rewrite_options,
            summary=summary,
            review_id=review_id,
        )

    def _self_check_rewrites(self, options):
        """对替代文案做禁用词自检，剔除仍然不合规的方案"""
        safe_options = []
        for opt in options:
            new_hits = self._matcher.match(opt.text)
            if not new_hits:
                safe_options.append(opt)
            # 如果仍有命中但风险为 LOW，保留但标注
            elif opt.risk_level == RiskLevel.LOW:
                safe_options.append(opt)
        return safe_options if safe_options else options  # 兜底：全部返回

    def _build_summary(self, hits, judgments, violations, citations, risk_level):
        """用确定性逻辑组装摘要，不额外调用 LLM"""
        violation_count = len(violations)
        hit_count = len(hits)
        categories = list({h.category for h, _ in violations})
        if risk_level == RiskLevel.HIGH:
            return (
                f"文案存在 {violation_count} 处明确违规（共命中 {hit_count} 个禁用词），"
                f"涉及类别：{'、'.join(categories)}。"
                f"已生成 {len(violations)} 个替代方案。"
            )
        elif risk_level == RiskLevel.MEDIUM:
            return (
                f"文案存在 {violation_count} 处疑似违规，置信度较低，"
                f"建议人工复核。涉及类别：{'、'.join(categories)}。"
            )
        return "文案基本合规。"
```

## 5. 错误处理与降级策略

| 失败场景 | 降级策略 | 用户体验 |
|---------|---------|---------|
| 禁用词典加载失败 | 抛出 `RuntimeError`，服务不可用 | 启动时检测，快速失败 |
| LLM 语境判断超时/异常 | `batch_judge` 内 catch，该 hit 默认标记为违规（`is_violation=True, confidence=0.0`） | 宁可误报，不漏报 |
| vmem 检索无结果 | 返回空 `citations`，跳过法规引用 | 结果中显示"未找到相关法规条款" |
| vmem 服务不可用 | catch 异常，返回空 citations + 标记 `rag_available=False` | 审查继续，但注明法规检索不可用 |
| LLM 替代文案生成失败 | 返回空 `rewrite_options` + 提示"自动生成失败，请手动修改" | 不阻塞主流程 |
| 替代文案自检不通过 | 兜底返回所有方案（见 `_self_check_rewrites`） | 标注"请人工确认" |
| DeepSeek API 限流 | `tenacity` 重试 3 次，指数退避（1s, 2s, 4s） | 延迟增加但不失败 |
| 结构化输出解析失败 | 重试一次，prompt 中加 "请严格按 schema 输出" | 对用户透明 |

补充重试装饰器的使用方式：

```python
        ...  # 如上
```

| 约束 | 对齐方式 |
|------|---------|
| LangChain 0.2+ | 使用 `langchain_openai.ChatOpenAI`，`with_structured_output()` 绑定 Pydantic schema |
| DeepSeek-V4-flash | `ChatOpenAI(base_url="https://api.deepseek.com/v1", model="deepseek-v4-flash")` |

| Pydantic 结构化输出 | `ComplianceReviewResult` 为顶层 schema，所有 LLM 调用点都使用 `with_structured_output()` 强制输出 |

整个架构的核心设计原则是：**确定性逻辑硬编码，推理逻辑交给 LLM，两者清晰分离。** 禁用词匹配和自检是确定性步骤（5ms），语境判断和替代文案是推理步骤（500ms-1s）。总延迟约 2-3 秒，其中 LLM 调用占 95%。

## 2. Data Import Pipeline

Three-stage flow in `parsers.py`:

**Law articles (`parse_law_articles` + `parse_internet_ads_rules`):**
- Regex splits on `第.{1,3}条` boundary
- Extracts article text + `【释义】` interpretation
- Auto-maps article number to chapter (e.g., 第九条 -> 第二章 广告内容准则)
- Auto-extracts semantic tags from content (e.g., contains "最高级" -> tag "极限用语")

**Penalty cases (`parse_cases`):**
- Splits on `### 案例 N` boundary
- Extracts structured fields via regex: title, violation content, law basis, penalty result, industry, violation type
- Each field stored in both readable text and `metadata` dict

**Industry rules (`parse_industry_rules`):**
- Splits on `## 一、行业名` boundary
- Preserves full content (law basis table + prohibited rules table + violation examples)

## 3. Retrieval Strategy (`ThreeLibRetriever`)

Four-step process:

**Step 1 -- Query routing:** `QueryRouter.classify_intent()` scores each library via keyword matching. Example: "化妆品广告永久美白会怎么处罚" hits `ad_case` keywords ("处罚") and `ad_industry` keywords ("化妆品"), giving higher weights to those libraries.

**Step 2 -- Parallel search:** Each library calls `store.search_fusion()` with its own `source_filter`. The fusion formula is:
```
score = w_vec * cosine_sim + w_fts * keyword_hit_rate + w_time * time_decay + w_conf * confidence
```
Default weights: `w_vec=0.5, w_fts=0.2, w_time=0.1, w_conf=0.2`

**Step 3 -- RRF merge:** Reciprocal Rank Fusion combines results across libraries:
```
final_score = 0.6 * RRF_score + 0.4 * normalized_original_score
```
where `RRF_score = sum(1 / (60 + rank_i))` across all lists a result appears in.

**Step 4 -- Feedback boost:** Results previously accepted by users get 1.5x multiplier; rejected results get 0.3x; modified get 0.5x.

**Context assembly (`build_retrieval_context`):** Orders results by priority: law articles first, then industry rules, then penalty cases, then banned words, then replacements. Caps at 6000 chars.

---

## 4. Feedback Memory Pipeline (`FeedbackPipeline`)

Three feedback types with confidence adjustment:

| Action | Storage | Confidence delta | Retrieval effect |
|---|---|---|---|
| `accepted` | `feedback:accepted:<ts>:<target_key>` | +0.1 (cap 1.0) | 1.5x boost |
| `rejected` | `feedback:rejected:<ts>:<target_key>` | -0.3 (floor 0.1) | 0.3x penalty |
| `modified` | `feedback:modified:<ts>:<target_key>` | -0.15 (floor 0.1) | 0.5x penalty |

Each feedback record stores: action type, target key, original query, user comment, original/modified text. Tags include `target:<key>` for precise retrieval during feedback-aware search.

```typescript
  request<FeedbackResponse>("/api/feedback", {
    method: "POST",
    body: JSON.stringify(body),

  request<StatsResponse>("/api/stats");
```

### 4. 状态管理 -- 审查流程状态机 `src/hooks/useReviewMachine.ts`

```typescript
import { useReducer, useCallback } from "react";
import type { ReviewPhase, ReviewResponse, Violation } from "@/types/review";

// ---- State ----
interface ReviewState {
  phase: ReviewPhase;
  text: string;
  industry: string;
  platform: string;
  result: ReviewResponse | null;
  selectedViolation: Violation | null;
  error: string | null;
}

// ---- Actions ----
type ReviewAction =
  | { type: "SET_TEXT"; payload: string }
  | { type: "SET_INDUSTRY"; payload: string }
  | { type: "SET_PLATFORM"; payload: string }
  | { type: "SUBMIT" }
  | { type: "SUBMIT_SUCCESS"; payload: ReviewResponse }
  | { type: "SUBMIT_ERROR"; payload: string }
  | { type: "SELECT_VIOLATION"; payload: Violation | null }
  | { type: "FEEDBACK_START" }
  | { type: "FEEDBACK_SUCCESS"; payload: string }  // updatedText
  | { type: "FEEDBACK_ERROR"; payload: string }
  | { type: "COMPLETE" }
  | { type: "RESET" };

const initialState: ReviewState = {
  phase: "idle",
  text: "",
  industry: "general",
  platform: "",
  result: null,
  selectedViolation: null,
  error: null,
};

/*
 * 状态转移图：
 *
 *   idle ──SUBMIT──→ submitting ──SUCCESS──→ result ──FEEDBACK──→ feedback ──SUCCESS──→ result
 *    │                  │                      │                                        │
 *    │                  │ ERROR                 │ SELECT_VIOLATION                       │ COMPLETE
 *    │                  ↓                      ↓                                        ↓
 *    │               idle(error)        selectedViolation                           completed
 *    │                                                                                   │
 *    └──────────────────────────────────RESET─────────────────────────────────────────────┘
 */
function reviewReducer(state: ReviewState, action: ReviewAction): ReviewState {
  switch (action.type) {
    case "SET_TEXT":
      return { ...state, text: action.payload };
    case "SET_INDUSTRY":
      return { ...state, industry: action.payload };
    case "SET_PLATFORM":
      return { ...state, platform: action.payload };
    case "SUBMIT":
      return { ...state, phase: "submitting", error: null };
    case "SUBMIT_SUCCESS":
      return { ...state, phase: "result", result: action.payload };
    case "SUBMIT_ERROR":
      return { ...state, phase: "idle", error: action.payload };
    case "SELECT_VIOLATION":
      return { ...state, selectedViolation: action.payload };
    case "FEEDBACK_START":
      return { ...state, phase: "feedback" };
    case "FEEDBACK_SUCCESS":
      return {
        ...state,
        phase: "result",
        result: state.result
          ? { ...state.result, highlightedText: action.payload }
          : null,
      };
    case "FEEDBACK_ERROR":
      return { ...state, phase: "result", error: action.payload };
    case "COMPLETE":
      return { ...state, phase: "completed" };
    case "RESET":
      return initialState;
    default:
      return state;
  }
}

export function useReviewMachine() {
  const [state, dispatch] = useReducer(reviewReducer, initialState);

  const setText = useCallback(
    (v: string) => dispatch({ type: "SET_TEXT", payload: v }), []
  );
  const setIndustry = useCallback(
    (v: string) => dispatch({ type: "SET_INDUSTRY", payload: v }), []
  );
  const setPlatform = useCallback(
    (v: string) => dispatch({ type: "SET_PLATFORM", payload: v }), []
  );
  const submit = useCallback(
    () => dispatch({ type: "SUBMIT" }), []
  );
  const submitSuccess = useCallback(
    (r: ReviewResponse) => dispatch({ type: "SUBMIT_SUCCESS", payload: r }), []
  );
  const submitError = useCallback(
    (msg: string) => dispatch({ type: "SUBMIT_ERROR", payload: msg }), []
  );
  const selectViolation = useCallback(
    (v: Violation | null) => dispatch({ type: "SELECT_VIOLATION", payload: v }), []
  );
  const feedbackStart = useCallback(
    () => dispatch({ type: "FEEDBACK_START" }), []
  );
  const feedbackSuccess = useCallback(
    (text: string) => dispatch({ type: "FEEDBACK_SUCCESS", payload: text }), []
  );
  const feedbackError = useCallback(
    (msg: string) => dispatch({ type: "FEEDBACK_ERROR", payload: msg }), []
  );
  const complete = useCallback(
    () => dispatch({ type: "COMPLETE" }), []
  );
  const reset = useCallback(
    () => dispatch({ type: "RESET" }), []
  );

  return {
    state,
    actions: {
      setText, setIndustry, setPlatform,
      submit, submitSuccess, submitError,
      selectViolation,
      feedbackStart, feedbackSuccess, feedbackError,
      complete, reset,
    },
  };
}
```

### 7. hooks 详细设计

#### 7.1 `useReviewSubmit` -- `src/hooks/useReviewSubmit.ts`

```typescript
export function useReviewSubmit(machine: ReturnType<typeof useReviewMachine>) {
  const { state, actions } = machine;

  const handleSubmit = useCallback(async () => {
    if (!state.text.trim()) return;
    actions.submit();
    try {
      const result = await submitReview({
        text: state.text,
        industry: state.industry as Industry,
        platform: state.platform || undefined,
      });
      actions.submitSuccess(result);
    } catch (err) {
      actions.submitError(
        err instanceof Error ? err.message : "审查请求失败，请重试"
      );
    }
  }, [state.text, state.industry, state.platform, actions]);

  return { handleSubmit, isSubmitting: state.phase === "submitting" };
}
```

#### 7.2 `useFeedback` -- `src/hooks/useFeedback.ts`

```typescript
export function useFeedback(machine: ReturnType<typeof useReviewMachine>) {
  const { state, actions } = machine;

  const handleFeedback = useCallback(
    async (violationId: string, action: "confirm" | "modify" | "dismiss", replacement?: string) => {
      if (!state.result) return;
      actions.feedbackStart();
      try {
        const res = await submitFeedback({
          reviewId: state.result.reviewId,
          violationId,
          action,
          replacement,
        });
        actions.feedbackSuccess(res.updatedText ?? state.result.highlightedText);
        toast({ title: "反馈已记录", description: getFeedbackMessage(action) });
      } catch (err) {
        actions.feedbackError(
          err instanceof Error ? err.message : "反馈提交失败"
        );
        toast({ variant: "destructive", title: "提交失败" });
      }
    },
    [state.result, actions]
  );

  return { handleFeedback, isPending: state.phase === "feedback" };
}
```

```typescript
import { useEffect, useState } from "react";
import type { StatsResponse } from "@/types/review";

export function useStats() {
  const [data, setData] = useState<StatsResponse | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<string | null>(null);

  useEffect(() => {
    let cancelled = false;
    setLoading(true);
    getStats()
      .then((res) => { if (!cancelled) setData(res); })
      .catch((err) => { if (!cancelled) setError(err.message); })
      .finally(() => { if (!cancelled) setLoading(false); });
    return () => { cancelled = true; };
  }, []);

  return { data, loading, error };
}
```

## 技术可行性验证报告

### 验证点 1: LangChain + DeepSeek-V4-flash 的 Function Calling 兼容性

**结论: RISK**

**分析:**

方案中使用 `ChatOpenAI(base_url="https://api.deepseek.com/v1", model="deepseek-v4-flash")` 来调用 DeepSeek，并期望 `with_structured_output()` 正常工作。这里有三个层面需要验证:

1. **Function Calling 基础兼容性:** DeepSeek API 兼容 OpenAI 协议，支持 `tools` 参数和 `tool_choice` 参数。DeepSeek-V3 起已支持 function calling。但方案中提到的 "deepseek-v4-flash" 是一个具体模型名，需要确认该模型是否在 DeepSeek API 中存在且支持 function calling。截至当前已知信息，DeepSeek 官方 API 提供的模型包括 deepseek-chat、deepseek-reasoner 等，"deepseek-v4-flash" 这个确切模型名需要在 API 文档中核实。

2. **`tool_choice` 可控性:** 方案声称 "tool_choice 可控"。DeepSeek API 确实支持 `tool_choice: "auto"/"none"/"required"` 以及指定具体函数名。但在实际使用中，DeepSeek 的 tool_choice 行为与 OpenAI 有细微差异——当 `tool_choice="required"` 时，DeepSeek 偶尔会在不需要工具调用的场景下强行调用。方案的编排层是硬编码流程（非 AgentExecutor），不依赖 LLM 自主决策 tool_choice，所以这个差异影响不大。

3. **`with_structured_output()` 的实现路径:** LangChain 的 `with_structured_output()` 有两种模式:
   - **原生模式 (method="json_schema"):** 需要模型原生支持 JSON Schema 约束输出。OpenAI 原生支持，但 DeepSeek 是否原生支持 `response_format: { type: "json_schema", schema: {...} }` 需要验证。
   - **工具模式 (method="function_calling"):** 将 Pydantic schema 包装为一个 tool，通过 function calling 的方式让模型输出结构化参数。DeepSeek 支持此路径。

   **风险点:** 如果 DeepSeek 不支持原生 JSON Schema 输出，LangChain 会回退到 function calling 模式。这要求 schema 被包装成一个"伪工具"，模型需要"调用"这个工具来输出结果。方案代码中直接使用 `llm.with_structured_output(ContextJudgment)` 而未指定 `method`，LangChain 会自动选择最佳路径，但具体行为取决于 DeepSeek 的 API 响应。

**修复建议:**
- 在 `ChatOpenAI` 初始化时显式指定 `model="deepseek-chat"` 或确认 DeepSeek API 中 "deepseek-v4-flash" 的确切可用性。
- 建议显式指定 `method="function_calling"` 以避免自动检测的不确定性: `llm.with_structured_output(ContextJudgment, method="function_calling")`。
- 添加一个启动时的连通性测试: 发送一条简单消息验证 API key、模型名、function calling 支持是否正常。

### 验证点 2: Pydantic structured_output 在 LangChain 中的使用方式

**结论: PASS (有条件)**

**分析:**

方案中使用 `llm.with_structured_output(ContextJudgment)` 的模式是 LangChain 0.2+ 的标准用法，语法正确。具体验证:

1. **Schema 兼容性:** `ContextJudgment`、`RegulationCitation`、`RewriteOption` 都是简单的 Pydantic BaseModel，只包含基础类型 (str, bool, float, list[str])。这与 LangChain 的 structured output 要求完全兼容。

2. **嵌套 Schema:** `ComplianceReviewResult` 包含 `list[BannedWordHit]`、`list[ContextJudgment]` 等嵌套列表。LangChain 的 structured output 支持嵌套 Pydantic 模型，但需要注意:
   - `BannedWordHit.position` 使用了 `tuple[int, int]` 类型。在 JSON Schema 转换时，tuple 类型会被映射为 `{"type": "array", "items": [...]}` 或 `{"type": "array"}`。部分 LLM 对 tuple 的生成不够稳定，可能输出 list 而非固定长度的 tuple。
   - `RiskLevel` 使用了 `str, Enum`，这在 JSON Schema 中会被映射为 `{"enum": ["high", "medium", "low", "pass"]}`，这是完全兼容的。

3. **多工具场景:** 方案中 `ContextJudge`、`RegulationRAG._rerank`、`RewriteGenerator` 各自创建独立的 `with_structured_output()` 实例。这是正确的——每个调用点绑定自己的输出 schema，互不干扰。

4. **`RerankOutput` 和 `RewriteOutput` 的局部定义:** 在 `regulation_rag.py` 和 `rewrite.py` 中，方案将 `RerankOutput(BaseModel)` 和 `RewriteOutput(BaseModel)` 定义为函数内部的局部类。这在 Python 中是合法的，LangChain 也能正常处理，但需要注意: 如果使用 pickle 序列化或异步任务分发，局部类可能有问题。当前方案使用 `asyncio.gather` 做并发，不涉及序列化，所以没有问题。

**修复建议:**
- 将 `BannedWordHit.position` 的类型从 `tuple[int, int]` 改为 `list[int]`，或使用 `conlist(int, min_length=2, max_length=2)` 来确保 LLM 输出稳定的 JSON array。否则 LLM 生成时可能输出 `[0, 5]` (list) 而非 tuple，Pydantic v2 会自动转换，但 Pydantic v1 可能报错。
- 将 `RerankOutput` 和 `RewriteOutput` 提升到 `schemas.py` 中定义，避免潜在的序列化问题。

### 验证点 3: vmem EmbeddingEngine 输出维度与 LangChain Retriever 接口兼容性

**结论: RISK**

**分析:**

方案中 vmem_client.py 声称底层使用 "bge-small-zh-v1.5 嵌入，512 维向量"。但查看现有的 `pipeline/config.py`:

```python
@dataclass
class EmbeddingConfig:
    model_name: str = "text2vec-base-chinese"  # 中文 embedding 模型
    batch_size: int = 32
    max_length: int = 512
```

**存在模型不一致:** 现有管线使用的是 `text2vec-base-chinese` (768 维)，而方案声称使用 `bge-small-zh-v1.5` (512 维)。需要明确选择哪个模型。

此外，方案的核心问题是: **vmem 不是 LangChain 的 Retriever**。方案中的 `VmemClient` 是一个自定义封装，不继承 `langchain_core.retrievers.BaseRetriever`。这意味着:

1. **不能直接用于 LangChain 的链式调用:** 如果未来想用 `RetrievalQA` 或 `create_retrieval_chain`，需要将 `VmemClient` 适配为 `BaseRetriever` 子类。
2. **当前方案绕过了 LangChain 的 Retriever 接口:** `RegulationRAG` 直接调用 `self._vmem.search()`，不经过 LangChain 的 retriever 抽象。这是可行的，但失去了 LangChain 生态的缓存、回调、追踪等能力。
3. **embedding 维度兼容性:** 如果 VmemStore 内部使用 `embedding_engine.encode()` 生成向量，那么只要同一个 engine 用于写入和查询，维度就是一致的。`search_fusion` 方法接受 `query_emb` 参数，所以外部编码的向量可以直接传入。

**修复建议:**
- 统一 embedding 模型选择。`bge-small-zh-v1.5` 在中文语义检索上优于 `text2vec-base-chinese`，建议在方案和现有管线中统一为 `bge-small-zh-v1.5` (512 维)。如果选择 `bge-small-zh-v1.5`，需要确认 `max_length` 限制: bge-small-zh-v1.5 的最大序列长度为 512 tokens，对于长法规条文可能需要截断。
- 建议将 `VmemClient` 实现为 `langchain_core.retrievers.BaseRetriever` 的子类，以便未来与 LangChain 生态集成:
  ```python
  from langchain_core.retrievers import BaseRetriever
  from langchain_core.documents import Document
  
  class VmemRetriever(BaseRetriever):
      vmem_client: VmemClient
      namespace: str
      
      def _get_relevant_documents(self, query: str) -> list[Document]:
          results = asyncio.get_event_loop().run_until_complete(
              self.vmem_client.search(query, namespace=self.namespace)
          )
          return [Document(page_content=r.value, metadata={"key": r.key, "score": r.score}) for r in results]
  ```
- 如果不走 LangChain Retriever，当前方案也能工作，但需要在 `VmemClient` 中自行实现重试、缓存、超时等逻辑。

### 验证点 4: 四个工具的调用顺序——死循环或无限推理

**结论: PASS**

**分析:**

方案的核心设计原则是"确定性编排"——不使用 AgentExecutor 的 ReAct 循环，而是硬编码 `if-else` 流程。这从根本上消除了死循环风险。

逐项验证:

1. **Step 0 (禁用词匹配) -> Step 1 (语境判断):** 单向依赖，Step 0 输出 `hits` 列表，Step 1 消费。无循环。

2. **Step 1 (语境判断) -> Step 2 (法规检索):** 单向依赖，Step 1 输出 `violations`，Step 2 消费。无循环。

3. **Step 2 (法规检索) -> Step 3 (替代文案):** 单向依赖，Step 2 输出 `citations`，Step 3 消费。无循环。

4. **Step 3 (替代文案) -> Step 4 (自检):** 单向依赖，Step 3 输出 `rewrite_options`，Step 4 用 `BannedWordMatcher.match()` 做纯确定性检查。无循环。

5. **Step 4 自检后的兜底逻辑:** `_self_check_rewrites` 方法中，如果所有替代方案都不通过自检，兜底返回原始方案列表。这不会触发重新生成——没有回路到 Step 3。

6. **并发调用安全性:** `batch_judge` 使用 `asyncio.gather` 并发调用 N 个 LLM 请求。`gather` 的 `return_exceptions=True` 确保单个请求失败不会取消其他请求。无死锁风险。

7. **无限推理防护:** 方案不使用 LLM 决定调用哪个工具，所以不存在"LLM 决定再次调用同一工具"的循环。LLM 的调用次数是确定的: 最多 N (语境判断) + 1 (法规重排) + 1 (替代文案生成) = N+2 次，其中 N = 命中禁用词数量。

**补充说明:** 虽然没有死循环，但存在一个性能风险: 如果命中大量禁用词 (比如 N=20)，`batch_judge` 会并发发起 20 个 LLM 调用。DeepSeek API 有并发限制 (通常 3-5 QPS)，可能导致大量请求被限流。方案中使用了 `tenacity` 重试，但 20 个并发请求同时触发限流重试可能导致雪崩。

**修复建议:**
- 在 `batch_judge` 中添加并发度限制，使用 `asyncio.Semaphore`:
  ```python
  async def batch_judge(self, text, hits, product_category=None, max_concurrent=5):
      semaphore = asyncio.Semaphore(max_concurrent)
      async def limited_judge(hit):
          async with semaphore:
              return await self.judge(text, hit, product_category)
      tasks = [limited_judge(hit) for hit in hits]
      return await asyncio.gather(*tasks, return_exceptions=True)
  ```
- 建议设置 `max_concurrent=3` 以匹配 DeepSeek API 的典型 QPS 限制。

### 验证点 5: 错误处理策略的 failure mode 覆盖率

**结论: RISK**

**分析:**

方案列出了 8 种失败场景及降级策略，覆盖面较好。但存在以下遗漏:

**已覆盖 (PASS):**
- 禁用词典加载失败 -> RuntimeError 快速失败: 合理。
- LLM 语境判断超时 -> 默认标记违规: 正确的降级策略。
- vmem 检索无结果 -> 返回空 citations: 可接受。
- vmem 服务不可用 -> catch 异常继续: 可接受。
- LLM 替代文案生成失败 -> 返回空 rewrite_options: 可接受。
- 替代文案自检不通过 -> 兜底返回: 可接受。
- DeepSeek API 限流 -> tenacity 重试: 正确做法。
- 结构化输出解析失败 -> 重试一次: 可接受。

**遗漏的 failure mode (RISK):**

1. **输入文本为空或超长:** `ReviewRequest.text` 有 `min_length=1, max_length=5000` 的 Pydantic 约束，但这只在 FastAPI 入口层生效。如果直接调用 `agent.review()` 而不经过 API 层，约束不生效。建议在 `agent.review()` 方法开头添加显式校验。

2. **禁用词典文件不存在或 JSON 格式损坏:** `_load_dictionary` 方法使用 `open()` 和 `json.load()`，如果文件不存在会抛 `FileNotFoundError`，JSON 格式损坏会抛 `JSONDecodeError`。方案中说"抛出 RuntimeError"，但实际代码没有 try-catch 包装。建议显式捕获并包装为 `RuntimeError`。

3. **LLM 返回空内容或拒绝回答:** DeepSeek 在某些情况下可能返回空的 assistant message (例如内容安全过滤)。`with_structured_output()` 在收到空响应时会抛 Pydantic ValidationError。方案中"结构化输出解析失败 -> 重试一次"覆盖了这个场景，但建议在重试 prompt 中加入更强的约束。

4. **`batch_judge` 中部分成功部分失败:** 方案使用 `return_exceptions=True` 并对异常做降级处理，这部分是正确的。但没有处理"LLM 返回了合法 JSON 但内容不合理"的情况——例如 `confidence` 值超出 `[0,1]` 范围。Pydantic 的 `ge=0, le=1` 约束会捕获这种情况并抛 ValidationError，但这会被归类为"异常"而非"LLM 输出质量问题"。

5. **并发安全:** `BannedWordMatcher.reload()` 方法重新加载词典，如果在 `match()` 执行过程中调用 `reload()`，会导致 `_entries` 列表在迭代过程中被修改。建议使用读写锁或原子替换策略:
   ```python
   def reload(self):
       new_entries = self._load_entries(self._dict_path)
       self._entries = new_entries  # 原子替换引用
   ```

6. **review_id 碰撞:** `f"CR-{int(time.time())}-{uuid.uuid4().hex[:4]}"` 使用 UUID 前 4 位 (16 bit)，在高并发下碰撞概率极低但非零。如果需要绝对唯一，建议使用完整 UUID 或数据库自增 ID。

7. **vmem namespace 隔离:** 方案使用 `namespace="ad_regulations"` 隔离法规数据和用户记忆。但 `VmemStore` 的 `search_fusion` 方法没有 `namespace` 参数——它只有 `source_filter`。如果 vmem 的真实实现不支持 namespace，法规数据可能与用户记忆混在一起。需要确认 vmem 实际 API 是否支持 namespace 隔离。

**修复建议:**
- 在 `agent.review()` 入口添加输入校验。
- 在 `_load_dictionary` 中添加 try-catch 包装。
- 在 `batch_judge` 中限制并发数并添加输入校验 (Pydantic re-validation after LLM output)。
- 确认 vmem 的 namespace 支持情况，如果不支持则通过 `source_filter` 实现等效隔离。

### 验证点 1: Markdown 文件解析法规条文的正则表达式覆盖率

**判定: FAIL**

**具体问题:**

**1a. `【释义】` 标记检测失败 (严重)**

实际文件 `《中华人民共和国广告法》释义.md` 中的释义标记跨三行书写:

```
【
释义
】本条是关于广告法立法目的的规定。
```

而 `config.py` 第85行定义的 `interpretation_marker` 是单行字符串 `"【释义】"`。`parsers.py` 第117行的检查:

```python
if cfg.interpretation_marker not in content and len(content) < 200:
    continue
```

永远不会在 content 中找到 `"【释义】"`，因此 interpretation 部分无法被识别。结果是: 释义文本全部被混入 `article_text` 字段，`interpretation` 变量始终为空字符串。数据不会丢失，但丧失了结构化区分能力。

**1b. 正文部分与释义部分重复入库**

文件有两段 `第X条` 模式: 正文部分 (行175-838，75条) 和释义部分 (行852+，75条)。正则 `r"(第.{1,3}条)\s*(.*?)(?=\n第.{1,3}条\s|\Z)"` 会匹配全部 282 次出现。正文部分因 `len(content) < 200` 被跳过 (约一半)，释义部分全部保留，最终产出约 140 chunks。同一文章在释义部分被完整保留，但其中的释义内容无法通过 `has_interpretation` 元数据识别。

**1c. page marker 残留干扰跨页条文**

文件中存在 `---`、`<!-- 第 N 页 -->`、孤立页码行 (如 `36`)。`strip_page_markers` 去除了 HTML 注释和孤立数字行，但 `---` 分隔线未被去除。当 `第X条` 的内容跨越分页时，正则 DOTALL 模式下 `.*?` 能正确跨越这些标记，但这些残留文本会混入 chunk 的正文。

**1d. 条文号解析的边界情况**

正则 `第.{1,3}条` 对中文数字的匹配: 一条到九十九条 (1-2个字符)、一百条 (3个字符，即"一百")。广告法共75条，均在覆盖范围内。互联网广告管理办法有15条，同样覆盖。此项无问题。

**修复建议:**

```python

interpretation_pattern = re.compile(r"【\s*释义\s*】")

# 或在 clean_text 中增加预处理:
def clean_text(text):
    text = re.sub(r"【\s*\n\s*释义\s*\n\s*】", "【释义】", text)
    ...
```

同时在 `strip_page_markers` 中加入 `text = re.sub(r"\n---\n", "\n\n", text)` 去除分隔线。

### 验证点 3: search_fusion 权重参数合理性

**判定: RISK**

**权重配置:** `w_vec=0.50, w_fts=0.20, w_time=0.10, w_conf=0.20`，总和 1.0。

**合理之处:**
- 向量权重最高 (0.50) 适合中文语义匹配场景
- 时间衰减最低 (0.10) 合理，因为法规数据相对静态，30天半衰期 (storage.py 第180行) 对 hackathon 演示足够
- 置信度权重 (0.20) 给反馈信号留出空间

**风险项:**

**3a. FTS 评分实现过于粗糙**

`_compute_fts_score` (storage.py 第166-170行) 使用逐字符命中率:

```python
query_chars = set(query.replace(" ", ""))
hits = sum(1 for c in query_chars if c in text)
return hits / len(query_chars)
```

中文单字 `最`、`的`、`是` 等高频字几乎在所有文本中都会命中，导致 FTS 分数虚高。例如查询 "广告法第七条" 中的 6 个字符，几乎 100% 会命中任何包含中文的法规文本。这使得 FTS 通道的区分度极低，实际相当于 w_fts 被浪费。

**3b. 权重缺少调优依据**

配置中没有记录权重选择的原因或 A/B 测试结果。建议在 README 中增加权重调优说明或至少提供一个 grid search 的备选值。

**修复建议:**

FTS 评分应使用至少 bigram 匹配或 jieba 分词后的词级命中率:

```python
def _compute_fts_score(query, text):
    import jieba
    query_words = set(jieba.cut(query))
    text_words = set(jieba.cut(text))
    if not query_words:
        return 0.0
    hits = len(query_words & text_words)
    return hits / len(query_words)
```

1. 要么删除 `reviewing` 阶段（同步 API 设计），要么增加 `POLL_START` / `POLL_RESULT` action 实现轮询
2. 增加 `CANCEL` action + `AbortController` 集成
3. 增加 `RETRY` action，从当前 `text` 重新提交
4. `FEEDBACK_SUCCESS` 应同时更新 `violations` 数组（移除 dismissed 项，标记 confirmed/modified 项）

---

### 验证点 5: 审查结果实时展示是否需要 WebSocket

**判定: PASS** (但有 RISK)

方案选择同步 HTTP POST 是正确的。广告文案审查的计算时间取决于 LLM 推理速度，对于短文案（<500字）通常 2-8 秒，不需要 WebSocket。同步请求 + 前端 Skeleton 过渡是合理的方案。

**RISK**: 方案自相矛盾。
- API 设计明确写"后端同步返回结果（若审查耗时 >5s，前端展示 skeleton 过渡）"
- 但 `ReviewPhase` 定义了 `"reviewing"` 并在注释中说"SSE 轮询"
- 实际代码中没有任何 SSE 或轮询逻辑

如果后端审查时间可能超过 30 秒（复杂文案 + RAG 检索 + LLM 推理），同步 HTTP 请求可能超时。这时需要:
- 要么用 `timeout` 配置（fetch 默认无超时，需设置 AbortSignal.timeout）
- 要么改为异步模式: `POST /api/review` 返回 `taskId`，前端轮询 `GET /api/review/{taskId}`

**修复建议**:
1. 删除方案中的 SSE/轮询描述，保持同步模式的简洁性
2. 在 `fetch` 调用中添加 `AbortSignal.timeout(30000)`（30秒超时）
3. 或者如果审查时间不可预测，改为异步模式并实现轮询

### 补充发现的问题

**(a) XSS 风险 -- RISK。** `HighlightedText` 使用 `dangerouslySetInnerHTML` 渲染后端返回的 HTML。如果后端生成的 `highlightedText` 包含用户输入的原始文案（可能包含恶意脚本），存在 XSS 风险。方案未提及 HTML 净化措施。

**修复建议**: 在渲染前用 `DOMPurify.sanitize(html)` 净化 HTML，或确保后端对用户输入做了 HTML entity escaping。

**(b) `RelatedCase` 数据来源缺失 -- RISK。** `Violation` 类型中定义了 `relatedCases?: RelatedCase[]`，但后端检索层返回的是扁平的 case text，没有按 violation 关联案例。这需要 LLM 在审查推理时建立 violation -> case 的映射关系。

**(c) 移动端方案未细化 -- RISK。** 方案提到 `<768px` 使用 Tabs 切换输入/结果，但 `ReviewLayout` 的 props 定义只有 `children: React.ReactNode`，没有定义如何区分左右栏。需要一个 responsive 切换机制（如 `useMediaQuery` hook 或 CSS-only 方案）。

**(d) `useStats` 缺少 SWR/缓存能力。** 直接用 `useState` + `useEffect` 做数据请求，没有请求去重、缓存、重新验证等能力。对于看板页面，建议使用 SWR 或 TanStack Query。

```python
def _compute_fts_score(self, query: str, text: str) -> float:
    query_words = set(jieba.cut(query)) - {" ", "\n"}
    text_words = set(jieba.cut(text)) - {" ", "\n"}
```

#### 修复 9：生成 highlightedText 和 violation positions

```python
# backend/lib/html_highlight.py
import re

def generate_highlighted_html(original_text: str, violations: list) -> str:
    """
    根据 violations 的 startIndex/endIndex 生成带 <mark> 标签的 HTML。
    先按位置排序，再逐段拼接，对用户输入做 HTML 转义。
    """
    import html as html_lib

    sorted_v = sorted(violations, key=lambda v: v.start_index)
    parts: list[str] = []
    cursor = 0

    for v in sorted_v:
        # 非违规部分
        if cursor < v.start_index:
            parts.append(html_lib.escape(original_text[cursor:v.start_index]))
        # 违规部分：转义后包裹 <mark>
        escaped = html_lib.escape(original_text[v.start_index:v.end_index])
        parts.append(
            f'<mark data-vid="{v.id}" '
            f'class="bg-red-200 dark:bg-red-900 rounded px-0.5 cursor-pointer '
            f'underline decoration-wavy decoration-red-500">{escaped}</mark>'
        )
        cursor = v.end_index

    # 尾部
    if cursor < len(original_text):
        parts.append(html_lib.escape(original_text[cursor:]))

    return "".join(parts)
```

此函数对用户输入做 `html.escape()`，防止 XSS。`HighlightedText` 前端组件用 `dangerouslySetInnerHTML` 渲染时无需额外 DOMPurify。

### 2.6 降级策略汇总

| 失败场景 | 降级策略 |
|---------|---------|
| 禁用词典文件不存在或格式损坏 | `RuntimeError`，启动失败，快速报错 |
| DeepSeek API 连通性测试失败 | 启动时 lifespan 中检测，log warning 但不阻塞启动 |
| LLM 语境判断超时/异常 | `asyncio.Semaphore(3)` 限流 + `tenacity` 3 次重试 + 默认标记违规 |
| LLM 返回空/拒绝/格式错误 | Pydantic ValidationError 重试一次，再失败默认违规 |
| vmem 检索无结果 | 返回空 citations |
| vmem 服务不可用 | catch 异常，返回空 citations + log |
| 替代文案生成失败 | 返回空 rewrite_options |
| 替代文案自检不通过 | 兜底返回所有方案 |
| review_id 碰撞 | 使用完整 UUID hex[:8]（32 bit） |
| 输入文案为空/超长 | agent.review() 入口显式校验 |

```typescript
// src/hooks/useReviewSubmit.ts
export function useReviewSubmit(machine: ReturnType<typeof useReviewMachine>) {
  const { state, actions } = machine;
  const abortRef = useRef<AbortController | null>(null);

  const handleSubmit = useCallback(async () => {
    if (!state.text.trim()) return;
    actions.submit();

    const controller = new AbortController();
    abortRef.current = controller;

    try {
      const result = await submitReview({
        text: state.text,
        industry: state.industry as Industry,
        platform: state.platform || undefined,
      }, controller.signal);   // 传递 signal
      actions.submitSuccess(result);
    } catch (err) {
      if (controller.signal.aborted) return;  // 用户取消，不报错
      actions.submitError(
        err instanceof Error ? err.message : "审查请求失败，请重试"
      );
    }
  }, [state.text, state.industry, state.platform, actions]);

  const handleCancel = useCallback(() => {
    abortRef.current?.abort();
    actions.cancel();
  }, [actions]);

  return { handleSubmit, handleCancel, isSubmitting: state.phase === "submitting" };
}
```

#### ViolationCard 核心布局

```tsx
// src/components/review/ViolationCard.tsx
interface ViolationCardProps {
  violation: Violation;
  isSelected: boolean;
  onClick: () => void;
  onFeedback: (violationId: string, action: "confirm" | "modify" | "dismiss", replacement?: string) => void;
  isFeedbackPending: boolean;
}

export function ViolationCard({ violation, isSelected, onClick, onFeedback, isFeedbackPending }: ViolationProps) {
  const config = SEVERITY_CONFIG[violation.severity];
  const [showLaw, setShowLaw] = useState(false);
  const [modifyText, setModifyText] = useState("");

  return (
    <Card
      className={`${config.border} ${isSelected ? "ring-2 ring-primary" : ""} cursor-pointer transition-all`}
      onClick={onClick}
    >
      <CardHeader className="pb-2">
        <div className="flex items-center gap-2">
          <Badge className={config.color}>{config.label}</Badge>
          <span className="text-sm text-muted-foreground">{violation.category}</span>
          <span className="ml-auto text-xs text-muted-foreground">{violation.lawArticle}</span>
        </div>
      </CardHeader>
      <CardContent className="space-y-3">
        {/* 违规文本高亮 */}
        <p className="text-sm">
          {violation.text.slice(0, violation.startIndex)}
          <span className="text-red-600 font-semibold underline decoration-wavy">
            {violation.text.slice(violation.startIndex, violation.endIndex)}
          </span>
          {violation.text.slice(violation.endIndex)}
        </p>

        {/* 违规原因 */}
        <p className="text-sm text-muted-foreground">{violation.explanation}</p>

        {/* 替代建议 */}
        {violation.suggestions.length > 0 && (
          <div className="space-y-1">
            <p className="text-xs font-medium">替代建议：</p>
            {violation.suggestions.map((s, i) => (
              <SuggestionCard key={i} suggestion={s} onApply={() => onFeedback(violation.id, "modify", s.replacement)} />
            ))}
          </div>
        )}

        {/* 法条溯源 */}
        <Collapsible open={showLaw} onOpenChange={setShowLaw}>
          <CollapsibleTrigger className="text-xs text-primary underline">
            {showLaw ? "收起法条" : "查看法条溯源"}
          </CollapsibleTrigger>
          <CollapsibleContent>
            <LawTracePanel
              lawArticle={violation.lawArticle}
              lawContent={violation.lawContent}
              relatedCases={violation.relatedCases}
            />
          </CollapsibleContent>
        </Collapsible>

        {/* 反馈按钮 */}
        <FeedbackBar
          onConfirm={() => onFeedback(violation.id, "confirm")}
          onModify={(replacement) => onFeedback(violation.id, "modify", replacement)}
          onDismiss={() => onFeedback(violation.id, "dismiss")}
          isPending={isFeedbackPending}
        />
      </CardContent>
    </Card>
  );
}
```

### 4.2 评测维度权重

**Agent 评测 (30 题)：**

| 维度 | 权重 | 说明 |
|------|------|------|

| risk_accuracy | 25% | 风险等级判定是否准确 |
| law_citation | 25% | 法条引用是否正确 |
| suggestion_quality | 15% | 替代建议是否合理 |

**RAG 评测 (15 题)：**

| 维度 | 权重 | 说明 |
|------|------|------|
| key_recall | 45% | 是否检索到正确文档 |
| source_recall | 30% | 是否命中正确的子库 |
| precision | 25% | 返回结果中相关文档占比 |
